# RTF Dashboard — 학습과 소통(학소) 회의 도구

**회의: 2026-07-15(수). 8시간, 100명 이상, 대표 참석.**
공장 담당자들이 품목을 하나하나 점검하며 공급을 조정하는 회의. 사용자는 이 자료로 평가받는다.

> **작업 전 반드시 `docs/재고전망-스펙.md`를 읽을 것.** 검증된 데이터 규칙·계산식이 전부 거기 있다.
> 이 파일은 요약과 함정만 담는다.

## 실행

```
start.bat        # 로컬 서버(8787) + 결산자료 사전계산(--if-stale) + 브라우저
```
`index.html`을 `file://`로 직접 열면 `fetch`가 막혀 결산자료가 안 읽힌다. **반드시 start.bat.**

새 달 결산 파일을 넣으면 `tools/build-closing.js`의 `MONTHS`에 월을 추가하고 재실행.

## 회의 스토리라인

```
도입   회의체계 · 회의안건
1부    RTF판정(어디서 품절나나) → 공급원인(왜 나나)          품절방어
2부    재고전망(총괄장) → 과잉감축(어디서 줄이나)             재고절감
마무리 조정총괄 · 회의록
```

**재고전망이 총괄장이다.** 6월 결산 리뷰(왜 늘었나)로 회의를 열고, 그대로 하반기
3단 전망(원계획 → RTF조정 → 과잉감축)으로 이어진다. 화면 상단 3단 KPI는 공장 담당자가
판정을 누를 때마다 실시간으로 내려간다 — **그 하강폭이 회의의 성과.**

헤드라인은 공시기준, 세부 전개는 관리기준, 차액은 조정행으로 명시.

## 회의에서 쓸 사실

**6월 총재고 2,531 → 2,714억(+183억). 증가의 96%가 thynC 하나(+169억).**

그런데 6월 thynC는 — **구매입고 207.1억 / 판매 2.2억 / 코스트센터출고 19.6억.**
코스트센터출고는 매출이 아니라 샘플·데모 등 내부 비용처리다.
→ ***"207억어치 사서 2억 팔고 20억은 비용으로 털었다."*** 생산이 아니라 **구매 의사결정** 문제.

- **카트 비피 프로 8~11호(56억)**: 3평판 기준 **68~75개월치**. 6월 입고 0인데도 재고가 안 줄어듦. thynC보다 구조적으로 심각.
- **계획 없는 재고 432억** (제·상품 360 + BOM 미사용 자재 72). 이 회의로는 못 줄인다 → 사업부에 소진 계획 요구.
- **불용 72억**: BOM에도 없고 6개월 무출고. 에포시스프리필드시린지 반제품 6종이 44억.
- **리브레**는 재고가 늘었지만 3평판 2.7개월 → **정상 성장. 감축 대상 아님.**

## 코드 지형

- `js/screens/rtf.js` — `buildHierarchy()`가 **유형>품목군>품목 4단 트리 + 펼침 + 정렬**을 이미 제공. 다른 화면의 트리는 이걸 재사용할 것 (`inventory.js`의 `buildInvHierarchy`는 품목 레벨 없는 축소판).
- `js/screens/excess.js` — **과잉감축의 판정·액션 구조는 이미 완성됨. 손대지 말 것.**
  - `AI_SECTION_DEFS`: ①입고·생산 축소 ②판매계획 현실화 ③재고 소진(마케팅·KPI 미반영) ④재고 처분
  - `AI_DECISION_LABELS`: 수용/조정 수용/불가/보류. **확정(수용·조정)만 KPI 반영.**
  - 감축 한도 = `min(적정초과분, 그 달 공급량, 이후 모든 월 최소 기말재고)` → **품절 0건 수학적 보장.**
  - **자재 반작용**: 완제품 감축 → 소비 감소 → **자재 재고 증가(+)**. `computeMatScenarioDeltas`가 정직하게 집계. 순 절감을 내려면 `computeAiMatExcessPlan(fgAdj)`로 자재 입고계획을 취소해야 함.
- `js/screens/summary.js` — 회의안건 차트(`daysTagsPlugin`이 재고일수 뱃지). 재고전망 차트가 이 스타일을 따른다.
- `js/screens/meeting.js` — **디자인 톤의 기준.** 딥네이비 히어로 `#1a3558` + 하늘색 액센트 `#7dd3fc` + 흰 카드 + 큰 타이포.

### 알려진 문제
- CSS 10개 파일에 **하드코딩 색 1,910개 / 고유 379개** (constraint.css 142, excess.css 95, rtf.css 72). 토큰 통일 필요.
- `inventory.css`: `.inv-chart-section`이 42행(260px)과 261행(400px)에 **중복 정의**. `.inv-toolbar`도 84·265행 중복.
- `renderExcessFgTable`(excess.js:552) — 그룹형 트리 버전이 정의만 되고 **호출되지 않음**(flat으로 교체된 흔적).

## 작업 방식

- **확인 전 코딩 금지.** 설계는 먼저 합의한다.
- **팩트 기반.** 추정치를 실측처럼 말하지 않는다. 예시 숫자는 예시라고 밝힌다.
- 회의가 코앞이므로 **한 화면을 부분만 바꾸지 않는다** (특히 색 토큰 — 전 화면 동시 적용).
